# 速く出荷する方法：シリコンバレーで身につけた習慣

シリコンバレーには多くの問題がある。住宅危機。モノカルチャー。妄想に傾くこともある楽観主義。

でも、一つのことは深く正しくやっている：速度を個性ではなく、規律として扱う。

それを学ぶのに4年かかった。残ったものを共有する。

## 1. コードを書く前に、コミットメッセージを書く

重要な機能を始める前に、最終的に届けたいコミットメッセージを先に書く。「マジックリンクによるユーザー認証を追加——パスワードリセットフローを廃止し、サポートチケットを削減」

これは計画の儀式じゃない。2分でできる。実装に迷い込む前に*成果*を明確にするための強制だ。開始前にきれいなコミットメッセージを書けないなら、まだ自分が何を作るかわかっていない。

## 2. 恥ずかしいくらいの状態で出荷し、素早く反復する

何でも最初のバージョンは恥ずかしいものであるべきだ。壊れているじゃなく——恥ずかしい。v1を誇りに思えるなら、出すのが遅すぎた。

この区別は重要だ：恥ずかしいとは「もっとうまくできるが、学ぶには十分だ」という意味。壊れているとは「まだ動いていない」という意味。前者は特性で、後者は遅延だ。

## 3. スコープを削る。品質を削らない

締め切りが厳しいとき——常にそうだが——本能はすべてを雑にやることだ。テストカバレッジを減らす。エッジケースを飛ばす。「TODO: 後でちゃんと直す」とコメントを書いて次に進む。

より良い動き：スコープを削る。7つのことを中途半端にやるより、3つのことを完全にやる。ユーザーは品質に気づく。そこにあるものが機能すれば、何が欠けているかにはほとんど気づかない。

## 4. フィードバックループをできる限り短くする

自分のラップトップの設定は、「何かを変えた」から「結果を見る」までの時間を短縮することだけを目的に最適化されている。ホットリロード。リアルなシードデータを持つローカルデータベース。保存時に走る自動テスト。

フィードバックを待つ1秒は、文脈を失う1秒だ。バレーで一緒に働いた最高のエンジニアたちのフィードバックループは、分単位ではなくミリ秒単位で測られていた。

## 5. すべてをタイムボックスする

タスクが1日以上かかるなら、それぞれ1日未満のタスクに分割する。分割できないなら、まだ十分に理解していない。理解はスピードの前提条件だ。

## 6. 1行書く前に、ユーザーデータを読む

ユーザーが実際にやっていることを見ずに作ったすべての機能は、間違った機能だった。例外なく、毎回。

ユーザーデータは不快だ。なぜなら、自分の仮定が間違っていたことを示すから。まず仮定を修正する。それから作る。

## 注意点

方向のない速度は、ただの移動だ。間違った方向に速く進んでしまうことを見てきたし、自分もやってきた。上の習慣は、正しい問題を向いているときだけ意味を持つ。

正しい問題を見つけることはそれ自体がスキルで、それはもっとゆっくりした仕事だ。そこにショートカットはない。
